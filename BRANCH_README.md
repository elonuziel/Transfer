Branch README — Transfer (changes vs main)

Purpose

This file documents the primary changes and upgrades introduced on this branch compared to the main branch. It is intended for reviewers to quickly understand what changed, why, and how to test locally.

High-level summary

- Ephemeral long-polling in-browser chat (no persistence) for quick local collaboration.
- Auto-close/watchdog: server shuts down after ~5 minutes idle to conserve battery.
- Ktor server: ActivityTrackerPlugin, in-memory circular buffer for chat (last 100 messages), and safer long-polling handling.
- Web UI: chat Floating Action Button (FAB), overlay chat panel, unread badge, responsive styling and reconnection logic.
- Security: Long-polling chat GETs excluded from activity timer so silent pages don't keep the server awake; existing IP approval & password protection preserved.

Files changed (most relevant)

- app/src/main/kotlin/.../KtorServer.kt
  - Adds: chat endpoints (/api/chat GET long-polling, POST to post), ActivityTrackerPlugin, watchdog coroutine.
- app/src/main/java/.../FileServerService.kt
  - Handles ACTION_STOP_SERVICE to gracefully stop the service when watchdog triggers.
- app/src/main/assets/index.html
  - Adds FAB and chat panel markup and bindings.
- app/src/main/assets/style.css
  - Chat panel and bubble styles; mobile/responsive rules.
- app/src/main/assets/script.js
  - Long-polling loop, message buffering, sender persistence in localStorage, unread count handling.

Design & rationale

- Long-polling is chosen to avoid adding WebSocket infrastructure while still providing near-real-time UX and reduced battery/network use.
- Messages are intentionally ephemeral (memory-only) to keep the app privacy-friendly and minimize storage/complexity.
- The ActivityTrackerPlugin excludes long-polling GETs to prevent a passive open client from preventing the server from auto-stopping.

How to build & verify

1. Build: .\\gradlew assembleDebug (Windows) or ./gradlew assembleDebug (Unix).
2. Install debug APK to an Android device/emulator and start the FileServerService.
3. From another device on same network, open the app's served web UI, click the chat FAB, and send messages.
4. Verify messages appear across connected clients and that GET long-polling reconnects on timeout.
5. Leave UI idle; verify service stops after ~5 minutes (check logcat for ACTION_STOP_SERVICE).

Migration notes

- No DB migrations required; chat is in-memory and ephemeral.
- If persistent chat is desired, replace in-memory buffer with storage and add retention controls.

Next steps / Review checklist
